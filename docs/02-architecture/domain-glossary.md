# Domain Glossary

**Project**

Distributed Payroll Processing Engine

---

# Purpose

This glossary defines the ubiquitous language used throughout the project.

All documentation, source code, specifications, tests and architecture documents must use these terms consistently.

No synonyms should be introduced without updating this document.

---

# Company

A tenant within the platform.

Each company owns:

- Employees
- Payroll Periods
- Payroll Jobs
- Payroll Transactions
- Payslips

Every business operation belongs to exactly one Company.

---

# Employee

A worker employed by a Company.

Employees participate in payroll processing.

Each employee belongs to only one company.

---

# Payroll Period

A time interval for which payroll is calculated.

Examples:

- January 2026
- February 2026
- March 2026

A company may have only one active Payroll Job per Payroll Period.

---

# Payroll Job

Represents the request to process payroll for an entire company during a specific Payroll Period.

A Payroll Job is the orchestration unit of the platform.

It does not calculate payroll directly.

Instead, it coordinates distributed processing.

State transitions:

- PROCESSING
- COMPLETED
- FAILED

Business Rules:

- One Payroll Job per Company and Payroll Period.
- Must be idempotent.
- Generates one Payroll Transaction per Employee.

---

# Payroll Transaction

Represents the payroll processing of a single Employee.

Each Payroll Transaction belongs to one Payroll Job.

Payroll Transactions are processed independently.

State transitions:

- PENDING
- PROCESSING
- COMPLETED
- FAILED

Failures do not affect other Payroll Transactions.

---

# Payslip

Represents the final payroll result for an Employee.

Contains:

- Gross Salary
- Bonuses
- Deductions
- Taxes
- Net Salary

A Payslip is immutable after generation.

Corrections require a new business operation.

---

# Payroll Calculation

The business process responsible for calculating an Employee's payroll.

This project implements conceptual payroll calculations.

Country-specific legislation is intentionally outside the project scope.

---

# Payroll Processing

The distributed execution engine responsible for processing Payroll Transactions.

Responsibilities include:

- Parallel execution
- Retry handling
- Kafka consumption
- State management
- Idempotency
- Publishing business events

---

# Projection

A read model generated from business events.

Projections are optimized for queries.

They do not contain business rules.

Projections are eventually consistent.

---

# Command

An intention to change system state.

Examples:

- Create Payroll Job
- Update Employee Salary
- Retry Payroll Processing

Commands execute business rules.

---

# Query

A request for information.

Queries never modify business state.

Queries are served from read models whenever possible.

---

# Domain Event

A business event that represents something meaningful that already happened.

Examples:

- PayrollJobCreated
- PayrollProcessingStarted
- EmployeePayrollCompleted
- PayrollCompleted

Domain Events are immutable.

---

# Integration Event

An event published for consumption by other bounded contexts.

Integration Events enable asynchronous communication.

Every Integration Event originates from a Domain Event.

---

# Transactional Outbox

Pattern used to guarantee reliable event publication.

Business data and pending events are committed atomically.

An Outbox Publisher asynchronously publishes events to Kafka.

---

# Idempotency Request

Represents a previously executed command.

Contains:

- Idempotency Key
- Request Fingerprint
- Response
- Expiration

Ensures that identical commands execute only once.

---

# Idempotency Key

A unique identifier generated by the client.

Every critical command must include an Idempotency-Key.

Repeated requests using the same key must return the original response.

---

# Correlation ID

A unique identifier representing an entire business workflow.

The Correlation ID is propagated across:

- HTTP Requests
- Kafka Events
- Logs
- Audit Records

Used for distributed tracing.

---

# Optimistic Locking

Concurrency control mechanism based on entity versioning.

Every update verifies the expected version.

If another update already modified the entity, the operation fails.

The client must reload the latest state before retrying.

---

# Retry

Automatic reprocessing of failed operations.

Retries are limited.

Repeated failures are redirected to the Dead Letter Queue.

---

# Dead Letter Queue (DLQ)

Kafka topic containing messages that could not be processed successfully after exhausting all retry attempts.

DLQ messages require manual inspection or replay.

---

# Read Model

Materialized view optimized for query operations.

Read Models are built asynchronously from Kafka events.

They may temporarily lag behind the command side.

---

# Aggregate

A consistency boundary within the domain.

Business invariants must be enforced inside Aggregates.

Aggregates communicate through Domain Events.

---

# Bounded Context

A logical boundary that owns a specific business capability.

Each Bounded Context:

- owns its models
- owns its rules
- owns its persistence
- owns its APIs

Communication between contexts occurs only through public contracts.

---

# Specification

The formal definition of expected system behavior.

Specifications are the source of truth.

Implementation must always follow Specifications.

---

# ADR

Architecture Decision Record.

Documents why an architectural decision was made.

ADRs preserve architectural knowledge throughout the project.

---

# Definition of Done

A feature is considered complete only when:

- Specification is approved.
- Architecture is updated if necessary.
- ADR is created when required.
- Tests are implemented.
- Tests pass.
- Documentation is updated.
- Code review is completed.

Implementation alone is never considered "Done".
